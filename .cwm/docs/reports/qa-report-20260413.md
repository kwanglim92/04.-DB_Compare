## 발견 및 수정 내역

### [심각] SQL Injection — `_fetch_spec_items` 테이블명 직접 보간

- **파일**: `src/core/sync_manager.py` (구 line 334)
- **문제**: `table_name` 파라미터를 f-string으로 SQL에 직접 삽입. psycopg2는 식별자(테이블명/컬럼명)를 `%s` 바인딩으로 처리할 수 없으므로 문자열 포맷팅을 사용하지만, 외부에서 임의 값이 넘어올 경우 SQL injection이 가능하다. 현재 호출부(`_sync_profiles`)는 리터럴 문자열을 전달하지만, 함수 시그니처가 public에 준하는 `str` 타입이므로 향후 확장 시 위험.
- **수정**: `_ALLOWED_SPEC_TABLES = frozenset({'profile_overrides', 'profile_additional_checks'})` 화이트리스트를 클래스 변수로 선언하고, 함수 진입부에서 `ValueError`를 발생시키는 가드를 추가. 허가된 값만 보간되도록 보장.

---

### [경고] 예외처리 없는 설정 파일 읽기 — `load_settings`

- **파일**: `src/ui/main_window.py` (line 81)
- **문제**: `settings.json`을 `json.load()` 로 읽을 때 예외처리가 없음. 파일이 손상되거나 인코딩 오류가 발생하면 앱 시작이 unhandled exception으로 중단.
- **수정**: `try/except (json.JSONDecodeError, OSError)` 블록으로 감싸고, 실패 시 defaults를 유지하면서 경고 로그만 출력하도록 변경.

---

### [경고] 예외처리 없는 버전 파일 쓰기 — `_save_local_versions`

- **파일**: `src/core/sync_manager.py` (line 156)
- **문제**: `sync_versions.json` 저장 시 예외처리 없음. 디스크 공간 부족, 권한 문제 등 `OSError` 발생 시 상위 `sync()` 메서드 전체가 예외로 종료되고 `(False, message)` 반환 계약이 깨짐.
- **수정**: `try/except OSError` 블록 추가. 저장 실패 시 에러 로그만 남기고 sync 결과에는 영향 없도록 처리.

### [경고] 스타트업 동기화가 UI 메인 스레드에서 동기 실행 — `_initial_sync`

- **파일**: `src/ui/main_window.py` (line 98)
- **문제**: `__init__` → `_initial_sync()` 순서로 호출되므로 서버 연결(최대 5초 timeout) 및 전체 데이터 다운로드가 메인 스레드를 블로킹. 앱 창이 표시되기 전에 수 초간 무응답 상태가 발생하고, 느린 네트워크에서는 UI가 실질적으로 멈춘 것처럼 보임.
- **현재 수정 여부**: 미수정 (로직 변경이 필요하므로 아래 구조 개선 제안 항목 참조)

---

### [참고] 암호화 키 파일 권한 설정 없음 — `CredentialManager._get_or_create_key`

- **파일**: `src/utils/credential_manager.py` (line 25)
- **문제**: `.key` 파일을 `Path.write_bytes()`로 생성할 때 OS 기본 umask가 적용됨. Windows AppData는 기본적으로 해당 사용자 전용이지만, 공유 PC 환경 또는 향후 Linux/macOS 이식 시 다른 사용자가 읽을 수 있는 권한(`644`)으로 생성될 수 있음.
- **현재 수정 여부**: 미수정. 운영 환경이 Windows AppData로 한정되어 있다면 즉각적인 위험은 낮음.

## 구조 개선 제안

### 1. 스타트업 동기화를 백그라운드 스레드로 분리

`_initial_sync()`를 `threading.Thread(target=..., daemon=True)` 로 실행하고, 완료 후 `self.root.after(0, callback)` 로 UI를 갱신하는 패턴을 사용하면 앱 시작 응답성이 크게 향상됨. `ServerSettingsDialog._test_connection`에서 이미 동일 패턴을 올바르게 사용하고 있으므로 참조 가능.

### 2. `get_config_dir()`의 mode 기본값 불일치

`config_helper.get_config_dir()`의 기본 파라미터는 `"offline"` 이지만, `load_settings()` 내부에서 `get_config_dir()` 를 인자 없이 호출하여 `settings.json`을 읽음. `sync_mode`가 아직 결정되기 전(생성자 초기 단계)이므로 의도적이나, 주석이 없어 독자가 혼동하기 쉬움. 명시적으로 `get_config_dir(mode="offline")`으로 작성하거나 주석 추가 권장.

### 3. `_sync_common_base` / `_fetch_spec_items` 코드 중복

두 메서드의 행 파싱 및 중첩 dict 구성 로직(module → part_type → part_name → list)이 거의 동일. 공통 헬퍼 `_rows_to_spec_dict(rows) -> Dict`로 추출하면 유지보수성이 향상됨.

### 4. `has_local_cache()`가 `common_base.json` 존재만 확인

`profiles/` 디렉토리 존재 여부나 프로파일 파일 수를 함께 확인하지 않아, common_base만 있고 profiles가 없는 반쪽짜리 캐시 상태에서도 `True`를 반환. `_on_server_settings_saved` 등에서 이 값을 상태 텍스트 판단에 사용하므로 오해를 유발할 수 있음.
