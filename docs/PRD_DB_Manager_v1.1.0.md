## 1. Executive Summary

DB_Manager는 **Park Systems XE Service의 장비 DB 품질검사(QC)를 자동화**하는 사내 전용 데스크톱 도구다. 기존 수작업으로 수행되던 2,600개 이상 DB 항목 검증을 **수분 내 자동화**하여 QC 엔지니어의 공수를 단축하고 휴먼에러를 제거한다.

v1.1.0까지는 **로컬 XML 기반 독립 검증 도구**로 완성되었으며(5일 집중 개발 → 2025-12-26 v1.0 릴리즈 → 2026-03-11 v1.1 Profile Manager 고도화), **차기 v1.2(Server DB Sync)** 에서는 중앙 서버(PostgreSQL) 기반으로 전환하여 전사 Spec 일관성을 확보한다. 이후 로드맵은 멀티-DB 비교, 히스토리 추적, 대시보드를 통해 **QC 운영 플랫폼**으로 확장한다.

### 핵심 비즈니스 임팩트 (추정)

| 지표 | As-Is (수작업) | v1.1.0 (현재) | v1.2 (예정) |
|------|---------------|--------------|-------------|
| 장비 1대당 QC 소요 시간 | 2~4 시간 | 10~15 분 | 10~15 분 |
| Spec 배포 주기 | 이메일/수동 | EXE 재배포 | **실시간 동기화** |
| 휴먼에러 발생률 | 체감 높음 | 근사 0 | 근사 0 |
| QC 엔지니어 공수 절감 | — | **약 85%** | **약 85% + 운영공수 감소** |

## 2. Product Overview (v1.1.0 현황)

### 2.1 제품 정의
Park Systems XE Service DB(XML 기반)를 파싱하여 사전 정의된 QC 프로파일과 비교하고, Excel 리포트를 자동 생성하는 Windows 데스크톱 애플리케이션.

### 2.2 기술 스택
- **언어/런타임**: Python 3.11+
- **UI**: CustomTkinter (Dark/Light 테마)
- **리포트**: OpenPyXL (Excel 3-시트)
- **배포**: PyInstaller 단일 EXE + NSIS 설치 프로그램
- **설정**: JSON (Common Base + Equipment Profiles 상속 구조)

### 2.3 현재 기능 요약 (v1.1.0)

| 영역 | 기능 |
|------|------|
| **DB 파싱** | XML 계층 구조 자동 파싱 (Module → Part Type → Part → Item) |
| **검증 엔진** | 3종 검증 타입: Range(Min/Max), Exact(일치), Check(기록) |
| **프로파일 관리** | Common Base 상속 구조, 듀얼 뷰(Table/Tree), 실시간 검색, CRUD |
| **Add Items from DB** | 기존/신규 항목 시각 구분, 실시간 카운트 |
| **QC 실행** | 트리뷰 컬러코딩 (Pass/Fail/Check), 요약 카운트 |
| **리포트** | 3-시트 Excel (Summary / All Items / Failed Items) |
| **누락 항목 리뷰** | N/A Items Review Dialog (DB 누락 자동 감지) |
| **보안** | 관리자 비밀번호 보호(`pqclevi`), 원본 DB 읽기전용 |

## 3. Problem Statement

### 3.1 현재의 문제
1. **수작업 QC의 비효율**: XE Service 장비 1대당 2,000개가 넘는 DB 항목을 엔지니어가 수작업으로 검토하며, 1대당 수 시간이 소요됨.
2. **휴먼에러 리스크**: 수작업 검증에서 발생하는 누락·오판정은 장비 출하 후 고객사 현장 이슈로 직결됨.
3. **Spec 관리의 파편화**: 장비별 QC 기준(Spec)이 엔지니어마다 다른 파일/기억에 산재되어 있음.
4. **Spec 배포 병목 (v1.1.0 한계)**: 현재는 Spec 변경 시마다 EXE를 재빌드·재배포해야 해서 긴급 수정이 어려움.

### 3.2 영향 범위
- **직접 영향**: XE Service QC 엔지니어 (주사용자), QC 팀 리드
- **간접 영향**: 장비 출하 품질 → 고객 만족도 및 A/S 비용

### 3.3 미해결 시 비용
- QC 엔지니어 인건비 및 출하 지연
- 필드 불량에 따른 **A/S 출동 비용** 및 **브랜드 신뢰 저하**

---

## 4. Strategic Goals

1. **QC 공수 절감**: 장비 1대당 QC 소요 시간을 수작업 대비 **80% 이상 단축** (유지·확장).
2. **품질 표준화**: Spec 중앙화를 통해 전사 QC 기준의 **단일 진실 공급원(Single Source of Truth)** 확립.
3. **출하 품질 지표 개선**: 출하 후 DB 설정 오류에 기인한 필드 이슈를 **전년 대비 50% 감소**.
4. **운영 효율화**: Spec 변경 시 EXE 재배포 없이 **실시간 반영** 가능 (v1.2 목표).
5. **이력 추적성 확보**: QC 결과의 **히스토리 저장 및 조회**를 통해 감사·트렌드 분석 기반 마련 (v1.3+).

---

## 5. Non-Goals

| 제외 항목 | 사유 |
|-----------|------|
| XE Service DB 이외의 타 제품군 지원 | 초기 ROI 불명확 — 별도 이니셔티브로 분리 |
| 실시간 협업 편집 (동시 다중 사용자 편집) | 사용자 규모(QC 엔지니어 수명)에 비해 복잡도 과다 |
| 외부 고객사 배포/SaaS화 | 사내 품질관리 도구이며 외부 공개 시 영업기밀 노출 리스크 |
| 장비 펌웨어·제어 기능 통합 | 제품 범위 초과 — QC 검증에 집중 |
| 모바일/웹 클라이언트 | 사용 환경이 엔지니어 PC 고정이며 ROI 낮음 |

## 6. User Personas

### P1. QC 엔지니어 (주사용자)
- **역할**: 출하 장비의 DB를 검증하고 리포트 제출
- **Pain**: 반복적·장시간 수작업, 휴먼에러 부담
- **기대**: 로드→실행→리포트의 원클릭 경험

### P2. QC 팀 리드 (관리자)
- **역할**: Spec 관리, 품질 지표 모니터링
- **Pain**: Spec 개정 시 배포 어려움, 히스토리 부재
- **기대**: 중앙 Spec 관리, 팀 단위 품질 대시보드

### P3. 현장 서비스 엔지니어 (부차 사용자, 미래)
- **역할**: 고객사 설치 현장에서 DB 설정 점검
- **기대**: 오프라인 환경에서도 동작, 간단한 리포트 공유

## 7. Key User Stories

**QC 엔지니어 관점**
- QC 엔지니어로서, DB 폴더 하나만 선택하면 **프로파일 기준으로 자동 검증**되기를 원한다 — 매번 수작업 비교를 하지 않기 위해.
- QC 엔지니어로서, 실패 항목만 필터링된 Excel 리포트를 즉시 받기를 원한다 — 우선 대응할 항목을 빠르게 파악하기 위해.
- QC 엔지니어로서, **DB에 누락된 항목(N/A)** 을 자동 감지받기를 원한다 — 빠트림 없이 출하하기 위해.

**QC 팀 리드 관점**
- 팀 리드로서, 모든 엔지니어가 **동일한 최신 Spec**을 사용하기를 원한다 — 버전 혼선을 방지하기 위해 (v1.2).
- 팀 리드로서, Spec 변경 시 EXE 재배포 없이 **실시간 반영**되기를 원한다 — 긴급 수정 대응을 위해 (v1.2).
- 팀 리드로서, 장비별·기간별 **QC 통과율 트렌드**를 보기를 원한다 — 품질 개선 지표 관리를 위해 (v1.3+).

**관리자 관점 (Spec 편집)**
- Spec 관리자로서, Common Base를 편집하여 모든 장비 프로파일에 공통 스펙을 상속시키기를 원한다 — 중복 관리 제거를 위해.

## 9. Technical Architecture (요약)

### 9.1 현재 (v1.1.0)
```
[DB Folder(XML)] → XMLParser → DBExtractor → QCComparator
                                                  ↑
                                          SpecManager (로컬 JSON)
                                                  ↓
                                     ChecklistValidator → ExcelReportGenerator
```

### 9.2 목표 (v1.2)
```
[사내 PostgreSQL :5434] ─(psycopg2)─→ SyncManager ─→ 로컬 JSON 캐시
                                                          ↓
                                                    SpecManager (인터페이스 동일)
                                                          ↓
                                      (이후 흐름 v1.1.0과 동일)
```
**설계 원칙**: Spec 저장소만 추상화되고 검증 엔진은 그대로 유지 → 구현 리스크 최소화.

## 12. Risks & Mitigations

| # | 리스크 | 영향 | 대응 |
|---|--------|------|------|
| 1 | 사내 PostgreSQL 서버 장애 | QC 중단 | **오프라인 폴백** (R10) — 로컬 JSON 캐시로 정상 동작 |
| 2 | 서버 접속 정보 유출 | 보안 사고 | 앱 내 **암호화 저장**, 관리자 비밀번호 보호 |
| 3 | Spec 변경 누락·실수 | 오판정 리스크 | v2.0 감사로그(R21) 및 RBAC(R22)로 통제 |
| 4 | 단일 개발자 의존 (Bus factor) | 유지보수 중단 위험 | 문서화 강화(PRD/CHANGELOG), 코드 리뷰어 확보 필요 |
| 5 | CustomTkinter 생태계 한계 | UI 확장성 제약 | v2.0 웹 대시보드로 UI 분리 검토 |
| 6 | DB 포맷 변경(XE Service 업데이트) | 파서 호환성 | Parser 추상화, 회귀 테스트 정비 |
